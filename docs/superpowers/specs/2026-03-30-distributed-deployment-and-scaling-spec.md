# 分布式部署与扩展 Spec

## 1. 目的

本文档定义目标平台的分布式部署原则。

目标：

- 支持多实例部署
- 支持按协议、站点、区域扩展
- 支持会话运行时、管理面、Agent、网关独立扩展

## 2. 核心组件

需要支持独立扩展的组件：

- `xuanwu-device-gateway`
- `xuanwu-management-server`
- `XuanWu`
- `xuanwu-iot-gateway`

## 3. 扩展策略

### 3.1 `xuanwu-device-gateway`

按以下维度扩展：

- 接入区域
- 会话量
- 设备类型

要求：

- 支持多实例
- 支持会话粘性或会话注册表

### 3.2 `xuanwu-management-server`

按以下维度扩展：

- API 读写压力
- 事件查询压力
- 遥测接入压力

要求：

- 配置真源共享
- 关系映射真源共享
- 事件与遥测存储共享

### 3.3 `XuanWu`

按以下维度扩展：

- Agent 推理量
- 编排压力
- 设备调用量

要求：

- Agent 真源共享
- 模型配置共享

### 3.4 `xuanwu-iot-gateway`

按以下维度扩展：

- 协议类型
- 站点
- 区域
- 厂商接入量

要求：

- 可按协议拆分部署
- 可按工业/家居/传感器场景拆分部署

## 4. 基础标识

以下 ID 必须全局唯一：

- `device_id`
- `user_id`
- `channel_id`
- `agent_id`
- `gateway_id`
- `trace_id`
- `event_id`
- `command_id`

## 5. 数据分层

建议至少分成：

- 配置真源层
- 关系映射层
- 事件与遥测层
- 运行时会话层

## 6. 高可用要求

- 管理面多实例
- 网关多实例
- 运行时多实例
- Agent 平面多实例
- 健康检查和故障摘除

## 7. 观测要求

所有组件至少输出：

- 结构化日志
- 指标
- trace_id
- request_id

## 8. 回压与故障隔离

- 网关故障不能拖垮管理面
- 某类协议故障不能拖垮全部网关
- 遥测洪峰不能拖垮控制 API
- Agent 推理波动不能阻塞设备注册

## 9. 最终结论

如果没有分布式部署规范，平台会很快在规模化阶段失稳。

只有把 runtime、management、agent、gateway 四层彻底拆开并支持独立扩展，平台才能真正走向工业化。
